# Grog - Vulnerability Management Platform

## 📋 О проекте

**Grog** — это современная платформа для управления уязвимостями (Vulnerability Manager), разработанная на Kotlin/Spring Boot как альтернатива DefectDojo. Проект решает проблему производительности и масштабируемости, предоставляя быстрое и надежное решение для команд безопасности и разработки.

### 🔍 Основная суть

Grog централизует управление результатами сканирования безопасности из различных источников (SAST, SCA, DAST, Secret Detection), позволяя командам эффективно отслеживать, приоритизировать и устранять уязвимости в своих продуктах.

### ✨ Ключевые возможности

- **CRUD Проектов (Products)** — управление продуктами и сервисами
- **CRUD Engagement** — организация тестов безопасности и сканирований
- **CRUD Findings** — работа с найденными уязвимостями
- **Интеграция с GitLab** — автоматическое создание задач и отчетов
- **Интеграция с Jira** — синхронизация уязвимостей с задачами
- **Гибкая работа с инструментами**:
  - SAST (Static Application Security Testing)
  - SCA (Software Composition Analysis)
  - DAST (Dynamic Application Security Testing)
  - Secret Detection
- **Парсеры отчетов** — поддержка форматов SonarQube, OSV Scanner, Dependency Check и других

## 👥 Целевая аудитория

### Портреты клиентов

#### 1. DevSecOps Инженеры
- **Задачи**: Автоматизация процессов безопасности, интеграция сканеров в CI/CD
- **Боли**: Необходимость ручного сбора отчетов из разных инструментов, сложность трекинга уязвимостей
- **Выгода от Grog**: Централизованное управление всеми находками, автоматизация через API и интеграции

#### 2. AppSec Инженеры
- **Задачи**: Анализ уязвимостей, приоритизация рисков, координация с разработчиками
- **Боли**: Разрозненные данные, отсутствие единой метрики безопасности
- **Выгода от Grog**: Единая панель управления, гибкая система тегирования и фильтрации, аналитика

#### 3. Разработчики
- **Задачи**: Устранение уязвимостей в коде и зависимостях
- **Боли**: Непонятные отчеты сканеров, сложность определения приоритетов
- **Выгода от Grog**: Четкие описания уязвимостей, прямая связь с Jira/GitLab, контекстная информация

## 📊 Сущности приложения

### Основные сущности

#### 1. Product (Продукт)
Представляет собой приложение, сервис или систему, в которой проводятся тесты безопасности.

**Атрибуты**:
- `id` — уникальный идентификатор
- `name` — название продукта
- `description` — описание
- `tags` — теги для категоризации
- `createdAt` — дата создания
- `updatedAt` — дата обновления
- `owner` — владелец продукта

**Связи**: Один ко многим с Engagement

---

#### 2. Engagement (Взаимодействие/Тестирование)
Представляет конкретное тестирование безопасности или сканирование в рамках продукта.

**Атрибуты**:
- `id` — уникальный идентификатор
- `product` — ссылка на Product
- `name` — название тестирования
- `type` — тип (SAST, SCA, DAST, Secret Detection)
- `status` — статус (Planned, In Progress, Completed, Cancelled)
- `startDate` — дата начала
- `endDate` — дата завершения
- `tool` — использованный инструмент (SonarQube, OSV Scanner, etc.)
- `reportFile` — путь к файлу отчета
- `createdAt` — дата создания
- `updatedAt` — дата обновления

**Связи**: Многие к одному с Product, один ко многим с Finding

---

#### 3. Finding (Уязвимость/Находка)
Конкретная уязвимость или проблема безопасности, обнаруженная в ходе тестирования.

**Атрибуты**:
- `id` — уникальный идентификатор
- `engagement` — ссылка на Engagement
- `title` — заголовок уязвимости
- `description` — подробное описание
- `severity` — критичность (Critical, High, Medium, Low, Info)
- `cwe` — идентификатор CWE
- `cvssScore` — оценка CVSS
- `location` — местоположение в коде
- `evidence` — доказательства (сниппеты кода, логи)
- `recommendation` — рекомендация по исправлению
- `status` — статус (Open, In Progress, Resolved, False Positive, Won't Fix)
- `jiraTicketId` — ID задачи в Jira
- `gitlabIssueId` — ID задачи в GitLab
- `createdAt` — дата создания
- `updatedAt` — дата обновления

**Связи**: Многие к одному с Engagement

## 🏗️ Архитектура приложения

### Текущая архитектура: Монолит с возможностью разделения

Grog построен как модульный монолит на базе **Spring Boot (Kotlin)**, что обеспечивает:
- Быструю разработку и развертывание
- Простоту тестирования
- Легкость будущего разделения на микросервисы

### Компоненты архитектуры

```
┌─────────────────────────────────────────────────────────────────┐
│                         Frontend                                │
│                    (React/Vue.js + TypeScript)                  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      API Gateway / Controller Layer             │
│                    (Spring MVC REST Controllers)                │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      Service Layer                              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────────┐ │
│  │ProductService│  │EngagementSvc│  │FindingService + Parsers │ │
│  └─────────────┘  └─────────────┘  └─────────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      Repository Layer                           │
│                    (Spring Data JPA Repositories)               │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────────────────┐
│ PostgreSQL  │    │    Redis    │    │   External Services     │
│  (Database) │    │   (Cache)   │    │ (GitLab, Jira, LDAP)    │
└─────────────┘    └─────────────┘    └─────────────────────────┘
```

### Будущее разделение на сервисы

При росте нагрузки монолит может быть разделен на:
1. **Product Service** — управление продуктами
2. **Engagement Service** — управление тестированиями
3. **Finding Service** — работа с уязвимостями и парсерами
4. **Integration Service** — внешние интеграции (GitLab, Jira)
5. **Parser Service** — обработка отчетов сканеров
